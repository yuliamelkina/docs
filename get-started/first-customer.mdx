## Overview

This guide walks you through creating customers, understanding KYC requirements, managing endorsements, and handling verification status.

## Required Information

To create an individual customer, you'll need:

### Personal Details
- Full name (first and last name)
- Email address
- Date of birth (must be 18+ years old)
- Residential address

### Identifying Information
- Social Security Number (for US customers)
- Government-issued ID details

### Legal Requirements
- Signed Terms of Service agreement ID
- Requested endorsements (payment rail permissions)

## Understanding Customer Status

Customer status indicates their verification and compliance state:

<Accordion title="Status Values">
- **`active`** - Fully verified and can use all approved endorsements
- **`under_review`** - Initial verification in progress
- **`awaiting_questionnaire`** - Additional information needed
- **`awaiting_ubo`** - Ultimate beneficial owner information required
- **`incomplete`** - Missing required information
- **`not_started`** - Customer created but verification not begun
- **`paused`** - Verification temporarily halted
- **`rejected`** - KYC verification failed
- **`offboarded`** - Customer account closed
</Accordion>

### Endorsement Status

Each endorsement has its own status:

- **`approved`** - Ready to use
- **`pending`** - Under review
- **`under_review`** - Being processed
- **`incomplete`** - Additional requirements needed
- **`rejected`** - Not approved
- **`revoked`** - Previously approved but now removed

## Address Requirements

### US Addresses
For US customers, use ISO 3166-2 subdivision codes with country prefix:

```json
{
  "residential_address": {
    "street_line_1": "123 Main Street",
    "street_line_2": "Suite 100",
    "city": "New York",
    "subdivision": "US-NY",
    "postal_code": "10001",
    "country": "USA"
  }
}
```

### Common US Subdivision Codes
- **US-CA** - California
- **US-NY** - New York
- **US-TX** - Texas
- **US-FL** - Florida
- **US-IL** - Illinois

## Managing Customer Information

### Retrieving Customer Details

Check customer status and requirements:

```bash
curl -X GET http://localhost:3000/customers/cust_alice_j_1234567890 \
  -H "Api-Key: test-api-key" \
  -H "Tenant-Id: your-tenant-uuid"
```

### Updating Customer Information

Update customer details when needed:

```bash
curl -X PUT http://localhost:3000/customers/cust_alice_j_1234567890 \
  -H "Api-Key: test-api-key" \
  -H "Tenant-Id: your-tenant-uuid" \
  -H "Idempotency-Key: $(uuidgen)" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "alice.newemail@example.com",
    "residential_address": {
      "street_line_1": "789 New Street",
      "city": "Los Angeles", 
      "subdivision": "US-CA",
      "postal_code": "90210",
      "country": "USA"
    }
  }'
```

## Best Practices

### Data Validation
- Always validate email addresses before submission
- Ensure birth dates indicate customers are 18+ years old
- Use correct ISO country and subdivision codes
- Verify SSN format for US customers

### Security
- Never log or store sensitive identifying information
- Use HTTPS in production environments
- Implement proper error handling for PII
- Consider data retention and deletion policies

### User Experience
- Collect Terms of Service acceptance before API calls
- Provide clear messaging about verification requirements
- Show progress indicators for endorsement status
- Handle pending and review states gracefully
